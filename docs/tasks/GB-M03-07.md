# GB-M03-07 - Successor preset, starter kit, and return to control

**Status:** In progress - the three-authority contract is fixed by accepted [`SPEC-CONFLICT-031`](../spec-conflicts/SPEC-CONFLICT-031-m03-successor-recovery-authority.md). Implementation remains route-gated until its protocol, atomic PostgreSQL writer, authenticated real-QUIC dispatch, native two-confirmation handoff, adverse/restart evidence, metrics, and audit pass.

## Objective

Turn one committed normal death into exactly one legal living successor without reconstructing death-time authority, duplicating identity or starter grants, exposing a partially initialized character, or interrupting recovery with commerce. The successor must reuse the dead hero's immutable Core preset and roster slot, enter Character Select preselected, and return to Hall/control in no more than two confirmations.

## Three design authorities

1. `Gravebound_Production_GDD_v1_Canonical.md`: `DTH-020`, `DTH-021`, `UI-007`-`009`, `TECH-021`-`023`, and `QA-101` require the primary Create Successor path, last class/appearance preset, nonplayable dead identities, at most two confirmations, no commercial interruption, durable retry, and measured recovery.
2. `Gravebound_Content_Production_Spec_v1.md`: `CONT-CATALOG-003` fixes the exact Grave Arbalist starter equipment, provenance, Red Tonic unit identities/placement, empty slots, and new-instance requirement.
3. `Gravebound_Development_Roadmap_v1.md`: `GB-M03-07` plus the M03 no-duplicate, death-to-control, successor-to-combat, 25-loop, and private-loop gates own successor recovery and its evidence boundary.

Accepted [`SPEC-CONFLICT-031`](../spec-conflicts/SPEC-CONFLICT-031-m03-successor-recovery-authority.md) fixes universal preset persistence, former-slot reservation, atomic starter reuse, durable identity/replay, selected-character publication, and the two-confirmation handoff without weakening accepted identity, item, death, or terminal contracts.

## Durable contract

- Every normal death stores one immutable successor preset inside the death transaction: account/death/character binding, former roster ordinal, class ID, typed Core base-silhouette snapshot, content revision, and canonical preset hash.
- The latest unconsumed death reserves its former ordinal. Ordinary Create cannot steal it; live roster selection remains read-only with respect to the reservation.
- One serializable writer creates the successor and every required aggregate participant, initializes the exact `04D` starter plan inside the same transaction, consumes the reservation, selects the successor, and persists immutable result/receipt/audit/outbox rows.
- The new character starts at level 1, XP 0, full 120 health, safe Character Select state, no Oath/Bargains, empty Armor/Charm/backpack/CharacterSafe/Belt slot 2, exact starter Weapon/Relic, and two distinct Red Tonic units in Belt slot 1.
- The dead identity stays archived with null playable ordinal; Memorial/Echo/history, account-safe currency, Vault, and all other living characters are unchanged.
- Exact retry returns the stored result before current validation. Altered mutation reuse, changed death, foreign account, stale content, consumed/superseded reservation, corrupt result, or slot conflict fails closed with a typed audited outcome.

## Append-only protocol contract

- Advance protocol `1.16` to `1.17`; preserve all existing bytes and append reliable bounded successor message kind `23` under `core_successor_v1`.
- The client supplies only a server-issued death ID plus a nonzero mutation ID and current negotiated revision. It cannot author the successor identity, ordinal, name, class, appearance, starter plan, item UIDs, provenance, selection, or aggregate versions.
- The result carries stored death/successor/receipt IDs, former ordinal, class/base silhouette, starter item UIDs, exact post-aggregate versions, selected-character identity, content revision, canonical result hash, and typed fresh/replay/conflict/rejection status.
- Feature absence, normal-route admission, malformed/oversized frames, wrong reliable channel, foreign selection, and unsupported revision fail before repository access.

## Native recovery contract

- Create Successor is enabled only from a durably acknowledged terminal summary with negotiated authority; it remains disabled while loading/pending and absent from Memorial.
- Keyboard, pointer, and gamepad activate the same semantic command. One exact request remains retained while pending; retry resends identical bytes.
- Stored success enters Character Select with the successor preselected. Play is the second confirmation and reaches Hall/control without rename, store, promotion, or a duplicated transition.
- The surface preserves accessibility focus, reduced-effects parity, minimum 1280x720 layout, and the existing death-summary hierarchy.

## Required verification

- Protocol: pin all `1.16` bytes; append only kind `23`; reject absent feature, malformed/oversized/zero IDs, wrong channel, and foreign death before writes.
- Domain: copy the exact committed class/base silhouette/former ordinal; derive a new character and four new starter UIDs; exact kit/provenance/slots; level 1/XP 0/full health/no Oath/Bargains.
- PostgreSQL: fresh success, byte-identical replay after response loss/pool/process restart, altered conflict, concurrent same-death writers, separate account/death writers, reserved-slot enforcement, foreign/nonfinal/already-consumed/content mismatch/corrupt result/database outage/serialization retry.
- Atomicity: inject rollback after character, progression, world, life, Oath, inventory/items, selection, result/receipt, audit, and outbox families; every branch writes all or none.
- Real QUIC: lose response then replay stored result, restart server, reject normal endpoint, and finish with zero connection/actor/task/transaction/lock residue.
- Client: disabled before durable acknowledgement and in Memorial, one pending frame, exact retry, no replay animation duplication, preselected Character Select, Play-to-Hall, no commerce before control, and pointer/keyboard/gamepad focus parity.
- Integrated: 25 death-summary-to-successor-control journeys; median under 15 seconds, p95 under 30 seconds; at least 70% enter successor combat within 120 seconds; no duplicate identity, grant, selection, receipt, or outbox result.
