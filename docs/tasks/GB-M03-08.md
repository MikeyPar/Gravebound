## Durable terminal contract

### Successful extraction

- The server accepts only an authenticated, selected, living character in an active danger lineage with a valid authored extraction source and no unresolved mutation.
- The request binds nonzero mutation/extraction identity, account, character, lineage, restore point, content revision, canonical request hash, and exact account/character/world/inventory/life versions.
- One serializable transaction locks account, character, restore root/lineage/world location, aggregate heads, pouch/wallet, and item rows in documented order before planning.
- Equipped/Belt units become `Safe` in their existing slots. Run-material pouch stacks credit the safe wallet atomically.
- RunBackpack indices are visited ascending; durable UIDs within a projected stack are unsigned ascending. Consumables merge Belt, CharacterSafe, then Vault ascending before using empty CharacterSafe, Vault, Overflow, then ResolutionHold locations.
- Accepted RunBackpack items are never deleted for capacity. PersonalGround remains unaccepted instance-local custody and is not extracted.
- The result stores normalized placements, material credits, exact pre/post aggregate versions, canonical result hash, terminal/receipt IDs, Hall destination, outbox/audit identity, and replay state.
- Restore-root/lineage closure and Hall-transfer admissibility publish only with the complete placement result.

### Emergency and LinkLost Recall

- Explicit Recall starts from a reliable authenticated intent. Completion remains server-generated after exactly 12 authoritative ticks; cancellation before completion is allowed.
- While channeling, movement is exactly 75%; fire, abilities, interaction, and consumables are rejected. Damage does not cancel. Lethal health on the completion tick wins.
- Explicit completion uses terminal kind `EmergencyRecall`; the 90-tick LinkLost completion uses `DisconnectRecovery` with the same inventory-loss writer and a stored trigger.
- Equipped/Belt units become `Safe` in place. All character-owned `AtRiskPending` items, including RunBackpack and PersonalGround, and every pouch stack are destroyed in canonical order with retained provenance and immutable ledgers.
- Recall preserves the living character, Oath, Bargains, Bell debt, progression, safe storage, and account state; closes the danger lineage/restore point; stops the combat clock; and commits Hall arrival.

### Overflow and ResolutionHold

- Append only `Overflow=8` and `ResolutionHold=9` after `Consumed=7`; never renumber existing durable discriminants.
- Overflow is account-owned, `Safe`, extraction-only, exactly 20 logical slots, and stores its 72-hour deadline. Automatic expiry/salvage remains disabled until `GB-M05-10`.
- ResolutionHold is extraction-owned, has no expiry, is not player-usable storage, and blocks danger entry plus unrelated inventory/crafting/gifting mutations.
- M03 exposes only bounded read, move-to-legal-storage, and explicit-destroy operations for held logical stacks. It does not add salvage, Forge, gifting, or manual Overflow/Hold deposits.

## Append-only protocol contract

- Advance the negotiated protocol from `1.14` to `1.15` without changing bytes `1`-`18`.
- Assign message kind `19` to a bounded extraction terminal frame and `20` to a bounded Recall intent frame; results remain reliable server events.
- Extraction echoes a server-issued extraction request ID and client mutation identity plus selected character, expected versions, issue time, and exact content revision. It cannot author a destination or result.
- Recall carries selected character, `Start`/`Cancel`, sequence, and client tick only. The server owns completion tick, terminal ID, trigger, loss plan, versions, and Hall result.
- Typed results distinguish accepted, pending, exact replay, altered-payload conflict, stale/foreign authority, unavailable source, unresolved mutation, terminal lost, storage resolution required, database unavailable, and corrupt stored authority.
- Compatibility fixtures retain immutable `1.14` death-view, `1.12` safe-inventory, and older protocol frames. Malformed, oversized, unknown, disabled-capability, and wrong-channel frames fail closed before repository access.
