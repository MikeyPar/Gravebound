# GB-M02-07 - Real-protocol headless journey bot

## Objective

Provide a reusable headless client that chooses intent only from protocol-visible state and proves movement, aim, combat, pickup, death, manual Emergency Recall, and reconnect through real QUIC, protocol `1.4`, and the existing managed-session authority boundary.

## Authorities

1. `Gravebound_Production_GDD_v1_Canonical.md`: `SIM-003`-`SIM-012`, `COM-001`-`COM-009`, `DTH-001`-`DTH-010`, and `TECH-010`-`TECH-015`.
2. `Gravebound_Content_Production_Spec_v1.md`: immutable `fp.1.0.0` supplies the arena, player loadout, movement, attacks, enemies, rewards, pickup reach, and inventory outcomes. The bot cannot contain alternate gameplay facts.
3. `Gravebound_Development_Roadmap_v1.md`: `GB-M02-07` requires a headless bot that moves, aims, fights, picks up, dies, Recalls, and reconnects. The M02 exit gate later runs sixteen bots for two simulated hours.
4. Repository `ADR-024-protocol-visible-journey-bot-and-manual-recall.md`.

## Required implementation

- Implement a bounded snapshot assembler and observation model inside `bot_client`. Bot policy may inspect only decoded handshake, snapshot, reliable-result, and lifecycle-result messages.
- Generate checked monotonic Input, Action, Control, and mutation identities. The bot sends intent only and never imports server, inventory, encounter, or authority state.
- Aim at protocol-visible living enemies, hold primary through ordinary server cadence, move toward an eligible personal pickup, and request pickup only when the snapshot-visible distance is within the protocol policy threshold. Server range and inventory rules remain final.
- Preserve bot decision state across a replacement QUIC connection. Reconnect uses the prior protocol session ID and never reconstructs gameplay locally.
- Complete manual Emergency Recall from reliable `RecallStart`: exactly `400 ms = 12` authority ticks; movement intent is capped at `75%`; primary, abilities, interaction, consumables, and pickup mutations are unavailable; damage does not cancel; death on the completion tick wins; explicit `RecallCancel` returns to ordinary play.
- Manual Recall and death are terminal alternatives, so acceptance uses separate protocol journeys from the same bot implementation rather than claiming both outcomes in one character life.
- Add deterministic bounded real-QUIC journey coverage suitable for CI. The final sixteen-bot/two-hour runner and evidence are owned by `GB-M02-08`, where instance scheduling, tick diagnostics, memory sampling, and clean population teardown exist.
- Redact authentication bytes and avoid account data in bot reports. Reports contain bounded counts, protocol/session facts, terminal outcome, and failure reason only.

## Bounds and ordering

- Snapshot assembly accepts at most the protocol maximum of 64 chunks and 32 entities per chunk, rejects inconsistent metadata/duplicates, ignores older completed sequences, and never merges snapshot sequences.
- Bot command sequences begin at one and fail closed before `u32` overflow.
- Death is observed from authoritative player health/alive state and lifecycle destination only; the bot never predicts or authors death.
- Recall simulation advances first on each tick. If health reaches zero on the completion tick, death commits and Recall does not.
- The bot may select a pickup from visible eligible, alive, uncollected `PersonalPickup` entities only. A typed server mutation result is required before recording collection.

## Explicitly deferred

- Multi-instance ownership, production scheduler, tick percentile collection, and clean population teardown: `GB-M02-08`.
- Four-human impaired playability remains a conjunctive M02 exit gate.
- Accounts, durable session restore, item/death transactions, and PostgreSQL: M03.
- Production certificates, authentication service, deployment admission, and cross-process orchestration: later service/deployment packages.

## Acceptance

- Unit tests cover chunk order/duplication/inconsistency/bounds, fixed-point steering, checked sequence exhaustion, pickup eligibility/distance, typed result application, and terminal-state finality.
- Manual Recall tests pin start, redundant start, cancel, exact tick 11/12 boundary, 75% steady-state movement, held-fire/ability/pickup suppression, damage non-cancellation, and death-before-Recall ordering.
- Real-QUIC tests prove handshake, Join, snapshot-driven movement/aim/fire, authoritative enemy damage, visible pickup approach, accepted mutation, state-preserving reconnect on a replacement transport, manual Recall routing to Lantern Halls, and a separate authoritative death trace.
- No journey test calls `AuthoritativeSession`, `AuthoritativeArena`, inventory, pickup, health, or lifecycle mutation methods from bot policy. Server-side assertions may inspect final authority only after the protocol journey.
- Networking CI and full workspace CI pass before README advances.
