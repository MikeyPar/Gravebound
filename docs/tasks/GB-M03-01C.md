# GB-M03-01C — Native Core character creation and select

**Status:** Implemented; focused automated gates pass. Parent-package closure requires the combined release, visual, and workspace evidence recorded in the milestone audits.

## Objective

Present the wipeable Core identity authority through a native, server-authoritative character creation and select surface. The client must use validated content/copy, show only the approved Grave Arbalist subset, remain honest about deferred systems, and never fall through into M02 combat.

## Three design authorities

1. `Gravebound_Production_GDD_v1_Canonical.md`: `UI-001`, `UI-002`, `UI-007`, `UI-008`, `LOOT-050`, `TECH-003`, `TECH-004`, `TECH-020`, `TECH-021`, and `QA-101`.
2. `Gravebound_Content_Production_Specification_v1.md`: `CONT-000` through `CONT-003`, the immutable `fp.1.0.0` class/ability records, localization validation, and the prohibition on incomplete Core promotion.
3. `Gravebound_Development_Roadmap_v1.md`: M03 objective and work package `GB-M03-01`; PostgreSQL remains owned by `GB-M03-02`/`GB-M03-11`.

Approved `SPEC-CONFLICT-004` decisions are part of this contract.

## Scope

- Separate native Core identity startup route using protocol 1.6 and real QUIC.
- Loading, authentication, roster loading, empty, creation, submitting, populated, selecting, selected, disabled, error, disconnected, and retry presentations.
- Authoritative two-slot roster; one legal level-1 Grave Arbalist; no oath, item power, Play route, editable name, or appearance entitlement.
- Keyboard-only create, cancel, select, and retry flow with visible action availability.
- Validated `en-US` copy and mechanically sourced class/ability details.
- Explicit wipe warning and development feature/content identity.
- Automated real create/select screenshot journey at 1280x720 and 1920x1080.

## Exclusions

PostgreSQL and durable restart state, Lantern Halls, combat admission, starter items, oath selection, death/memorial/successor flow, production identity, formal Core promotion, and any unapproved character customization.

## Acceptance criteria

1. A native client authenticates to the wipeable endpoint, creates one legal character, refreshes the authoritative roster, and selects it without fabricating persistence or control transfer.
2. Reconnect refreshes Core identity instead of opening a combat session; disabled/error/disconnected states remain within the identity surface.
3. Copy and class data come from validated content, unresolved placeholders fail, and the FP records remain byte-for-byte unchanged.
4. Focused protocol/server/client tests, strict Clippy, content validation, optimized build/runtime, both required visual resolutions, and combined workspace gates pass.
