# GB-M03-10 - Authenticated read-only support lookup

**Status:** Repository implementation and schema-correct mandatory disposable PostgreSQL role gate are complete; the hosted rerun evidence and final audit remain open.

## Objective

Provide a bounded operator-only lookup for exact character, item, and death identifiers without exposing a gameplay mutation path, raw credential, platform identity, network address, unrestricted scan, wildcard query, or localized reconstruction.

## Three design authorities

1. `Gravebound_Production_GDD_v1_Canonical.md`: `TECH-005` keeps support inside the modular-monolith boundary; `TECH-020` defines durable records; `TECH-030`, `TECH-050`, `TECH-120`, `TECH-122`, `TECH-124`, and `TECH-125` require staff authentication, least privilege, immutable audits, secret hygiene, and item/death reconstruction without direct database edits.
2. `Gravebound_Content_Production_Spec_v1.md`: `CONT-LOC-001` makes localized support copy a separately validated content concern. The lookup therefore returns stored durable IDs, numeric states, content IDs/revisions, digests, and versions only; it does not reconstruct localized prose.
3. `Gravebound_Development_Roadmap_v1.md`: `GB-M03-10` requires support/debug lookup for character, item, and death. `GB-M04-10` and the M04 exit gate clarify the forward contract: read-only, authenticated, audited, bounded by exact incident IDs.

## Delivered contract

- Dedicated `support_lookup` workspace crate with no dependency on `server_app` or gameplay writers.
- Opaque operator tokens are length-bounded, redacted from `Debug`, wiped on drop, domain-hashed, and compared without data-dependent early exit.
- Only active `ReadOnlyLookup` principals can reach the PostgreSQL lookup adapter.
- Requests contain one exact nonzero 16-byte character, item, or death ID; one exact nonzero request ID; one typed reason; and one uppercase bounded case reference. No account-wide enumeration, wildcard, substring, free-text, or caller-selected limit exists.
- PostgreSQL reads run in an explicit read-only transaction through exact-ID, 65-row-capped `SECURITY DEFINER` functions over security-barrier views. Public function execution is revoked; the support role receives only explicit function execution and cannot select the views or gameplay tables directly. The views omit authentication/platform/network data, raw mutation payloads, and localized presentation fields.
- Character results expose current durable identity/state/version plus the newest 64 world-transfer transitions.
- Item results expose current durable custody/version plus the newest 64 item-ledger transitions.
- Death results expose durable terminal authority, trace digest, versions, and the newest 64 death audit transitions.
- A 65th transition is used only to set `transitions_truncated`; it is never disclosed. Missing exact IDs return a typed `NotFound` result.
- Every result is withheld until an append-only audit insert succeeds. Audit records contain operator ID, request/target IDs, typed reason, bounded case reference, typed outcome, and disclosed row count; never the credential or free-form customer data.
- Migration `0066_read_only_support_lookup_v1.sql` adds only the audit table, its append-only guard, bounded index, approved security-barrier views, and six exact-ID lookup functions. Production database grants remain an environment/IaC responsibility: the support role receives `EXECUTE` on these functions and `INSERT` on the audit table only. Runtime binding rejects roles that can select the views/base tables or mutate gameplay/audit state.

## Explicit exclusions

- Account, Echo, transaction-ID, party, session, currency, entitlement, moderation, or purchase lookup; those expand under `GB-M04-10` and later packages.
- Any correction, restoration, entitlement write, moderation action, content disable, database console, or arbitrary SQL facility.
- Operator provisioning, MFA, role grants, token rotation, network exposure, and support UI. These belong to hosting/IaC and later security packages.
