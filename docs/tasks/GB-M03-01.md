## Task contract

**Task ID:** `GB-M03-01`

**Title:** Wipeable Core identity and Grave Arbalist character creation/select

**Milestone:** M03 — First complete private character life

**Feature IDs:** `GB-M03-01`, `UI-001`, `UI-002`, `UI-007`, `UI-008`, `TECH-021`, `CONT-002`

**Player-visible outcome:** A tester launches the Core client, authenticates with an explicitly wipeable test identity, sees an authoritative two-slot roster, creates one legal level-1 Grave Arbalist, returns to character select, and selects that living character. The interface never implies durable restart persistence.

**Internal delivery split:**

The parent package closes only when all three slices pass together.

## Exact state

- Slot capacity: 2.
- Class: `class.grave_arbalist` only.
- Level: 1.
- Oath: absent; unavailable until level 10.
- Life state: living.
- Location/security state: safe, character select.
- Equipped item power: absent until `GB-M03-04`.
- Last played: absent until the character actually enters control.
- Persistence label: wipeable test namespace; restart wipe remains intentional in this package.

Client transitions:

```text
Boot -> PatchCheck -> Authentication.loading -> Authentication.success
     -> CharacterSelect.loading -> CharacterSelect.empty
     -> CharacterCreation -> CharacterCreation.submitting
     -> CharacterSelect.success -> CharacterSelect.selecting
     -> CharacterSelect.selected

Any safe network screen -> Disconnected -> Retry -> Bootstrap -> safe roster state
```

Server transitions:

```text
AbsentAccount -> TestAccount(version=1, roster=[])
TestAccount -> CharacterCreated(version+1)
TestAccount -> CharacterSelected(version+1)
AbsentCharacter -> LivingSafe(level=1, class.grave_arbalist, oath=None)
```

No Hall, danger, dead, retired, memorial, or durable transition is legal.

## Network and mutation contract

- Preserve existing message-kind IDs and canonical M02 frame bytes; append discriminants only.
- Bootstrap uses reliable ordered Control; create/select use reliable ordered Mutation.
- No account data travels over datagrams.
- Account snapshots are bounded and include authoritative version, slot capacity, roster, and selected character.
- Character mutations carry mutation ID, expected account version, payload hash, issued time, and bounded payload.
- An identical repeated mutation returns its original result.
- Reuse of a mutation ID with a different payload rejects as `idempotency_conflict`.
- A stale version rejects and returns the current safe account snapshot.

Typed errors include `unauthenticated`, `production_namespace_forbidden`, `account_mismatch`, `character_not_found`, `character_not_owned`, `character_dead`, `class_disabled`, `appearance_unavailable`, `invalid_name`, `character_slot_full`, `state_version_mismatch`, `idempotency_conflict`, `payload_hash_mismatch`, `issued_at_invalid`, `content_mismatch`, `rate_limited`, and `service_unavailable`.

## Persistence and privacy boundary

- In-memory test adapter keyed by authenticated test-account ID, never display name.
- One account writer lock protects version, roster, selected character, and bounded idempotency results.
- Reconnect to the same server process restores the aggregate.
- Server restart intentionally wipes the aggregate.
- Repository command/result types remain reusable by the later PostgreSQL adapter; validation does not move into SQL.
- No raw credential, display name, character label, auth token, or platform identifier enters logs or internal telemetry events.

## UI and accessibility

- Every safe screen has Loading, Empty, Success, Disabled, Error, Disconnected, and Retry behavior where applicable.
- Submission disables duplicate input while keeping cancel/retry semantics explicit.
- Character cards show only authoritative or honestly deferred fields.
- Keyboard navigation and visible focus are required.
- Minimum effective text size is 14 px with 24 px reference safe margins.
- Verify at 1280×720 and 1920×1080.
- State meaning uses text/shape in addition to color; reduced-motion/flash settings remain respected.

## Tests required

**Unit:** slot capacity, Arbalist-only create, exact defaults, create/select transitions, deterministic injected IDs/clock, duplicate retry, conflicting retry, stale version, cross-account rejection.

**Protocol:** bounded round trips, stable old discriminants/bytes, version mismatch, payload/account binding, oversized field rejection before allocation.

**Integration:** real QUIC bootstrap/create/refresh/select, dropped-response retry creates once, reconnect restores process-local roster, distinct credentials isolate accounts, restart proves wipe, content mismatch rejects, FP journey remains runnable in explicit FP mode.

**Adversarial/failure:** forged IDs, slot race, concurrent stale creates, mutation-cache/rate bounds, malformed text fields, production namespace rejection, credential-redaction scans.

**Content validation:** existing FP bytes/hash unchanged; Core development target resolves exactly one class and no prototype items; no premature promotion record.

**Visual/manual:** every UI state, keyboard-only flow, both target resolutions, focus order, clipping/safe margins, reduced effects, warning-free optimized runtime.

## Acceptance criteria

1. One wipeable test identity can bootstrap, create exactly one legal level-1 Grave Arbalist per accepted mutation, display its authoritative two-slot roster, and select a living owned character.
2. Creation and selection are bounded, authoritative, state-versioned, retry-safe, and reject invalid, cross-account, conflicting, and stale requests without partial mutation.
3. Native UI, real-QUIC journey, strict content/protocol/workspace gates, release build, and inspected visual evidence pass while PostgreSQL remains explicitly absent.
