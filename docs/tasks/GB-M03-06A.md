## Bounded record and protocol contract

- `DeathEvent` stores namespace, death/account/character IDs, request hash, pre/post aggregate versions, commit time/tick, instance/region/room/content revision, typed cause, killer/pattern/attack IDs, raw/final damage, damage type, pre-hit health, fixed-point source position, statuses, Recall/network snapshot, trace reference, and ordered destruction references.
- `DeathSummarySnapshot` is one-to-one with `death_id` and stores approved Core hero label, class, level, optional Oath, ordered Bargains, `lifetime_ms`, final deed ID, ordered last-five damage entries, and exact `Lost`/`Preserved`/`Created` projections including Echo status.
- `MemorialRecord` is one-to-one with `death_id` and stores account, `death_at`, summary revision, and the Core default presentation snapshot. It stores no entitlement and no item roll.
- Combat trace and destruction entries have explicit schema revisions, contiguous zero-based ordinals, bounded typed identifiers, fixed-point coordinates, and immutable foreign keys. The repository rejects gaps, duplicate ordinals, noncanonical order, nonfinite/caller-authored values, unknown discriminants, and incompatible content revisions.
- The final-death receipt stores `(character_id, "permadeath-v1")`, canonical payload hash, original typed result, issue time, and authoritative versions. Replay is checked before mutable-state validation.
- Extend protocol `1.12` append-only to `1.13`; preserve existing message-kind bytes `1`-`17` and assign the death-view/request capability the next unused byte `18`. Existing enum variants and compatibility fixtures are never reordered.
- The negotiated V1 surface exposes only authenticated account-bound views: latest committed death, summary by an account-owned historical `death_id`, bounded Memorial pages, and bounded trace pages. No client message can initiate lethal resolution or author the new `death_id`, cause, trace, destruction list, destination, Echo eligibility, presentation strings, or authoritative versions.
- Migration `0031` also adds the normalized immutable inventory/Belt component for the existing danger-entry restore root. Accepted danger entry snapshots exact Equipped/Belt UIDs and versions, transitions their unchanged locations from `Safe` to `AtRiskEquipped`, and records deterministic ledgers in the same caller-owned world-flow transaction; this prerequisite must be restart- and rollback-safe before `06C` can destroy danger custody.
- Malformed, oversized, unavailable-feature, foreign-owner, stale, corrupt, or unknown-version frames fail closed and do not touch the repository. A summary result is unavailable until its transaction commit is acknowledged.
