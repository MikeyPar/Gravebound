# GB-M01-11 - Red Tonic belt use and feedback

## Task contract

**Feature IDs:** `GB-M01-11`, `SIM-003`, `SIM-010`, `COM-004`, `CONT-010`, `CONT-FP-002`, `CONT-FP-007`, `CONT-FP-010`
**Depends on:** `GB-M01-01B`, `GB-M01-02E`; creates health and inventory seams consumed by `GB-M01-05A` and `GB-M01-07A`
**Decision:** [ADR-008](../decisions/ADR-008-red-tonic-healing-and-belt-order.md)
**Content records:** `consumable.red_tonic`, `arena.prototype.bell_laboratory_01`
**Persistence:** none; all tonic, healing, cooldown, and prototype health state is local-run state and is destroyed on restart.

## Required behavior

- Strictly compile Red Tonic as belt stack cap `6`, restore `3,000 basis points` of max health, restore duration `12 ticks`, shared potion cooldown `60 ticks`, damage does not interrupt restore, and consumed on accepted use.
- Bind `consumable_1` through the replaceable action map with `Q` and gamepad West (`X` / Square) defaults. One physical press produces one monotonic sequence; short presses survive render/fixed-rate mismatch; blocked input remains suppressed until release; stale or overflow sequences fail closed.
- Consumable input is not buffered. Evaluate each new press exactly once on its authoritative tick. Rejected input is consumed as input but never consumes an item, starts cooldown, or queues a later use.
- Reject use with typed reason `NoTonic`, `FullHealth`, or `Cooldown`. In particular, full health rejects without consumption or cooldown. Check full health after any healing already due on the tick.
- On accepted use, atomically remove one Tonic from belt slot 1, capture the resolved max health, create a 12-tick restore schedule, and start the 60-tick shared potion cooldown.
- Apply no healing on the acceptance tick. Apply scheduled healing on the 12 subsequent authoritative ticks `T+1` through `T+12`.
- For restore tick `i` in `1..=12`, compute cumulative scheduled healing directly from captured max health using integer half-up math: `C_i = round_half_up(max_health * 3000 * i / (10000 * 12))`. Apply only `C_i - C_(i-1)` on that tick. This guarantees the scheduled final total is exactly `round_half_up(max_health * 3000 / 10000)` without drift or front-loaded remainder rules.
- Clamp current health to max health independently on each restore tick. Discard that tick's overheal permanently; do not carry it into later ticks. Continue the remaining schedule, so later damage may still be healed by later scheduled ticks. Damage never cancels or pauses the restore.
- Cooldown begins on accepted use and becomes ready exactly 60 tick indices later. Any new press while it is nonzero is rejected without consumption and cannot create a concurrent restore.
- Initialize every fresh or restarted First Playable run with two Tonics in belt slot 1. Restart cancels active restore and cooldown, destroys prior belt contents, and reconstructs the exact two-Tonic start state.
- When `item.prototype.charm.undertaker_knot` is equipped, resolve the exact authored override to `3,500` restore basis points and `75` shared-cooldown ticks while preserving the 12-tick restore duration and all other Red Tonic rules. Unequipping or restarting returns to the baseline definition.
- Use deterministic belt merge order for future pickups/refills: merge slot 1 first, then slot 2 only when slot 2 already contains Red Tonic. Respect cap 6 per slot and return any unplaced remainder to the caller. Backpack insertion, creation/rearrangement of slot 2, ground lifetime, and full capacity resolution remain `GB-M01-07A` seams; this ticket must not silently destroy overflow.
- Emit simulation-owned accepted, rejected, heal-tick, heal-completed, cooldown-ready, and belt-count state suitable for deterministic traces and presentation. The client renders belt count, cooldown, 12-tick progress, health delta, and typed rejection feedback, with a clear local-use pulse and audio cue that do not obscure hostile combat.

## Legal transitions

- Input: `Idle -> PressObserved -> Accepted | Rejected(NoTonic|FullHealth|Cooldown)`; every observed sequence is terminal and never buffered.
- Belt slot 1: `Count(2) -> Count(1) -> Count(0)` on accepted uses only; rejected use leaves count unchanged.
- Restore: `None -> Scheduled(12)` on acceptance; then `Scheduled(12..2) -> TickApplied -> Scheduled(11..1) -> TickApplied -> Completed` on `T+1..T+12`.
- Cooldown: `Ready -> Cooling(60) -> ... -> Cooling(1) -> Ready`; the press at the first Ready boundary may be accepted if all other preconditions pass.
- Health: `current < max -> min(max, current + scheduled_delta)` per restore tick. Clamped excess is discarded for that tick.
- Merge: `remainder -> slot1 free capacity -> existing-Tonic slot2 free capacity -> returned remainder`.
- Restart: `AnyRunState -> Destroyed -> FreshRun(health=max, slot1=2, restore=None, cooldown=0)`.

## Failure behavior

- Missing, duplicate, or wrong-kind Red Tonic record; nonexact First Playable values; zero duration/cooldown/cap/restore; restore above 100%; `damage_interrupts_restore=true`; `consumed_on_use=false`; invalid health (`current > max` or zero max); invalid belt count; stale/overflow input sequence; or arithmetic overflow fails closed.
- Use acceptance is transactional. Item consumption, captured max health, restore schedule, cooldown, and event emission all commit or none commit.
- Heal-tick arithmetic is checked. A failure cannot partially change health, progress, cooldown, or belt state.
- Presentation errors cannot consume Tonics, heal, alter cooldown, invent rejection reasons, or change restart state.
- Unplaced merge remainder must be returned explicitly. It cannot be discarded merely because `GB-M01-07A` backpack/ground handling is not implemented.

## Tests required

- Unit/content: exact `6/3000/400/2000/false/true` authored tuple; `400 ms -> 12 ticks`; `2000 ms -> 60 ticks`; reference, kind, and value-drift rejection.
- Input: Q/gamepad-West defaults, replaceability, one sequence per press, short-press retention, blocked-until-release, no buffering, stale/overflow rollback.
- Acceptance: full-health, empty-belt, and cooldown rejection do not consume or start/reset cooldown; accepted use atomically consumes one and creates exact state.
- Healing: no acceptance-tick heal; exact ticks `T+1..T+12`; cumulative half-up schedules for divisible and nondivisible max health; exact final scheduled total; per-tick overheal discard; later damage may receive only remaining ticks; damage does not cancel.
- Cooldown: exact 60-tick boundary; repeated presses cannot overlap restores; first eligible boundary obeys health and belt preconditions.
- Belt/restart: fresh slot 1 count 2; cap 6; merge slot1 then existing-Tonic slot2; explicit remainder; restart cancels restore/cooldown and recreates exactly two Tonics.
- Determinism/failure: repeated trace pins use/rejection ticks, belt count, cooldown, schedule deltas, applied healing, discarded overheal, and final health; invalid arithmetic/state paths roll back.
- Visual/manual: optimized semantic capture proves Q binding, one accepted use, exact belt decrement, visible 12-tick restore/cooldown, health change, rejection feedback, and unobstructed combat view with audio/runtime logs clean.
