## Durable model

- Preserve every existing security discriminant: `Safe=0`, `AtRiskEquipped=1`, `AtRiskPending=2`, `Destroyed=3`.
- Preserve every existing location discriminant and append only `CharacterSafe=5` and `Vault=6` after `Equipped=0`, `Belt=1`, `RunBackpack=2`, `PersonalGround=3`, and `Destroyed=4`.
- `CharacterSafe(index)` accepts only `Safe` equipment or consumable units at indices `0..7` and retains character custody.
- `Vault(index)` accepts only `Safe` equipment or consumable units at indices `0..159`, remains account-owned, and has no character custody while stored. The acting/source character remains present on immutable ledger events.
- Equipped/Belt/RunBackpack/PersonalGround continue to require an owned character. Destroyed rows retain their existing terminal shape. Do not reserve location shapes for later systems.
- Consumable units remain independent durable UIDs. A projected stack is homogeneous by template and contains at most six units. Unit planning uses ascending unsigned UID order.
- `accounts.state_version` is the optimistic account/Vault aggregate version. `character_inventories.inventory_version` remains the character-storage version. A cross-scope accepted mutation advances each affected aggregate once, regardless of unit count; every moved item advances once.

## Safe-Hall transfer contract

- The dormant server surface exposes three bounded commands: CharacterSafe to Vault, Vault to the selected character's CharacterSafe, and CharacterSafe to RunBackpack as the explicit deliberate-risk action required before danger entry.
- A command binds nonzero mutation identity, account, selected character, exact source slot/item projection, expected account and inventory versions, and canonical payload hash. It never accepts a client-authored destination, security state, item version, or placement map.
- Every transfer requires an owned living selected character at the exact Lantern Halls safe location, with no unresolved inventory mutation. Character Select, danger, other safe destinations, foreign accounts, and stale selections fail closed.
- Equipment moves to the lowest empty legal destination. Consumables merge into matching nonfull stacks by ascending index, then use lowest empty indices; split placement and unit ordering are deterministic.
- Vault withdrawal rejects atomically with `character_safe_full`. Safe deposit rejects atomically with `storage_full`. Deliberate risk rejects atomically with `run_backpack_full`.
- One normalized `safe_inventory_mutations` receipt and bounded placement rows store request hash, command/source binding, pre/post account and inventory versions, exact per-UID placement, result hash, outcome, and commit time. Exact retry returns the stored result before fresh-state validation; the same mutation ID with any changed payload is an idempotency conflict.
- One accepted transfer updates custody/location/security and item versions, appends exactly one immutable ledger transition per UID, advances each affected aggregate once, records the result, and commits as one serializable transaction. Rejected or injected-failure paths expose no partial move.

## Transaction and lock order

1. Validate all bounded identifiers, variants, counts, and hashes before opening mutation state.
2. Begin the serializable retry boundary and lock account first.
3. Check the replay receipt before current-version/source validation.
4. Lock selected/owned character and exact safe Hall location, then character inventory.
5. Lock every involved item row in ascending unsigned UID order.
6. Plan the complete move against locked slots before the first write.
7. Apply item custody/security/location/version changes and append deterministic per-UID ledger events.
8. Advance account/inventory versions exactly once, insert normalized placements and receipt, then commit.

Manual transfers and danger preflight use the same ordering and planner so they cannot deadlock or disagree about capacity.

## Danger-entry preflight hook

- The hook runs inside the existing dormant serializable world-flow transaction after replay, source, content, and expected-version validation but before lineage, restore-point, or instance identities are allocated and before world location changes.
- Visit CharacterSafe slots ascending and plan the complete automatic deposit into matching Vault stacks and lowest empty Vault indices. Partial deposit is prohibited.
- If CharacterSafe is empty, preflight is a no-op with no version increment.
- If all items fit, commit their item/ledger/aggregate changes in the same entry transaction, prove CharacterSafe empty, and capture the post-mutation account/inventory versions in the danger restore root.
- If capacity is insufficient, return typed `storage_full`/`StorageResolutionRequired` with no item, ledger, version, ID-allocation, restore-root, lineage, or location mutation. A prior explicit CharacterSafe-to-RunBackpack action remains AtRiskPending and permits later entry.
- The hook is not a separate transaction and cannot admit the normal route. Production Realm Gate interaction remains disabled until all owning gates pass.

- Foreign account/character, wrong Hall location, dead/unselected character, stale account/inventory/item version, source drift, corrupt cross-axis state, replay, and altered-payload conflict.
