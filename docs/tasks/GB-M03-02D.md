# GB-M03-02D - Durable death and memorial persistence

**Status:** In progress - schemas `0031`/`0032` establish and harden the death record family; forward-only `0033`/`0034` make the prerequisite danger-entry graph component-complete, exact-identity, clock- and Ash-authoritative. Schema-34 hosted proof, exact crash restoration, and typed death repository DTOs remain open with `GB-M03-06A`.

## Objective

Add the wipeable Core PostgreSQL records and transaction boundary required to preserve a committed permadeath, its combat evidence, deterministic destruction result, exact summary, and memorial across response loss and process restart. This slice owns persistence mechanics, not combat-rule selection, client presentation, successor creation, extraction, or Recall.

Approved [`SPEC-CONFLICT-009`](../spec-conflicts/SPEC-CONFLICT-009-m03-death-memorial.md) is binding: every normal committed permadeath creates one memorial; `death_id` is UUIDv7; `(character_id, "permadeath-v1")` is the final-death idempotency key; and qualifying Echo projection occurs inside the same SQL transaction.

## Persistence contract and invariants

- Migration `0031` is forward-only and additive within the wipeable Core namespace. It may replace existing restrictive checks only to admit the named `dead` life state and `permadeath` destruction reason; it must not renumber existing item security/location values or rewrite migrations `0001`-`0030`.
- Add immutable, namespace-bound records for `DeathEvent`, `DeathSummarySnapshot`, `MemorialRecord`, ordered combat-trace entries, ordered item/material destruction entries, the final-death receipt, and append-only audit/outbox events. Every child row is foreign-keyed to `death_id`; no accepted death row may exist without its one-to-one summary and memorial.
- Store stable typed IDs, fixed-point numbers, ticks, versions, content revision, and presentation keys. Never store localized prose as authority, reconstruct the summary from mutable current state, or copy item rolls into memorial/Echo records.
- Lock the account, character aggregate, inventory aggregate, all owned at-risk item rows, and the Echo account aggregate in one documented order. The transaction changes `Living -> Dead`, advances each affected version once, and commits every named record or none.
- Exact retry returns the stored terminal outcome. Reuse with changed canonical payload is an audited idempotency conflict. A second final death for the same character is impossible even under a distinct mutation ID.
- After commit, commands bound to the former character or inventory versions fail closed. A database error, serialization victim, malformed stored record, illegal at-risk location, or projector failure leaves the character living and the pre-death inventory observable.
- CharacterSafe and Vault rows are read only for the preserved projection and are never destroyed or reassigned. Existing item rows transition to `Destroyed(permadeath)` and retain identity/provenance; they are not deleted.
- Rollback documentation must state that schema `0030` can be restored only after proving there are no dead characters, death/memorial/Echo rows, or `permadeath` destruction records. No automatic lossy down migration is permitted.
