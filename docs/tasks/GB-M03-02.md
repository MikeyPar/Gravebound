# GB-M03-02 — PostgreSQL durable aggregate parent package

**Status:** Complete - `02A`-`02D`, `GB-M03-11`, `GB-M03-12`, and the owning item/Vault/death packages pass; see [`GB-M03-02-audit.md`](../milestones/GB-M03-02-audit.md).

## Objective

Replace wipeable process-local account state with transactional PostgreSQL persistence, then extend the same boundary to the item, vault, memorial, and ledger aggregates only when their authoritative domain packages are approved. The final parent package must survive server restart without duplicating or partially applying any durable mutation.

- `GB-M03-02A`: PostgreSQL architecture and reusable `GB-M03-11` infrastructure.

## Invariants

- PostgreSQL is the only production-behavior database. SQLite and in-memory success cannot satisfy persistence integration gates.
- `persistence` owns SQL transactions, migrations, snapshots, and ledger storage; it never owns combat or product validation rules.
- Server domain services retain validation and state transitions. Repository transactions provide row locking, optimistic version checks, atomic commit, and typed infrastructure failures.
- Every durable mutation has a stable identity, payload hash, original result, issue time, account binding, and expected aggregate version.
- Retrying identical input returns the original result; altered payload conflicts; stale versions return current safe state.
- Pre-Early-Access data uses an explicit wipeable namespace. No production/Steam namespace or permanence claim is introduced.
- Migrations are forward-only artifacts with a rehearsed rollback procedure. Content rollback never reinterprets stored data.
- Secrets and raw identities never enter source, migrations, logs, diagnostics, or test snapshots.

## Infrastructure acceptance criteria

1. A clean documented command starts pinned PostgreSQL, applies all migrations exactly once, reports readiness, runs isolated integration tests, and tears down without cross-test residue.
2. Concurrent account/character mutations use real PostgreSQL locking and prove exactly-once commit, optimistic-version rejection, retry replay, conflict rejection, and rollback on injected failure.
3. Restarting `server_app` against the same wipeable database preserves the committed `GB-M03-01` roster and selection, while a separately initialized namespace is empty.
4. CI cannot pass persistence gates without executing them against PostgreSQL; migrations, schema drift, rollback rehearsal, credential redaction, and warnings-denied workspace gates pass.

## Deferred until owning state machines

Item instances, item locations, vault capacity, extraction placement, death/destruction, memorial/Echo contents, currency balances, and ledger reason codes are named parent outcomes but must not be modeled ahead of `GB-M03-04`, `GB-M03-06`, `GB-M03-12`, and `GB-M03-13`.
