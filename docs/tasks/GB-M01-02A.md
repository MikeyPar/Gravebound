# GB-M01-02A — Mouse aim and held primary fire

**Status:** Completed and audited on 2026-07-10. See [`docs/milestones/GB-M01-02A-audit.md`](../milestones/GB-M01-02A-audit.md).

## Task contract

**Task ID:** `GB-M01-02A`  
**Title:** Mouse aim and held primary fire  
**Milestone:** M01 — First Playable  
**Feature IDs:** `GB-M01-02A`

**Player-visible outcome:** The Grave Arbalist aims a visible crossbow independently toward the mouse and fires a readable stream of narrow Pine Crossbow bolts while the left mouse button is held. Bolts obey the exact authored cadence, speed, and range even while the player moves or the camera follows.

**In scope:**

- Immutable simulation weapon data compiled from the checked-in Pine Crossbow and Grave Arbalist primary records.
- Renderer-independent aim, sequenced held-fire input, cooldown, projectile identity, projectile motion, range expiry, and shot events at fixed 30 Hz.
- Rebind-ready left-mouse default for `primary_fire` and a future menu/input-capture gate.
- Cursor-to-world-to-northwest-simulation aim conversion using the active camera.
- Independent crossbow rotation, owner-visible friendly bolt presentation, aim reticle, muzzle feedback, and live weapon diagnostics.
- Unit, deterministic, content, adversarial, visual, optimized runtime, and CI verification.

**Out of scope:**

- Projectile collision with walls, enemy hurtboxes, or other entities; damage application; pierce consumption; enemies; hit VFX; hit audio. These belong to `GB-M01-02B` and later combat tickets.
- Grave Mark, Slipstep, Stillness, alternate items, equipment mutation, controller/right-stick aim, aim-assist, networking, and persistence.
- Production sprites/audio. This step uses deliberate high-readability debug presentation.

**Authoritative owners:** `sim_core::weapon` and `sim_core::combat`; only validated `sim_content` may construct `WeaponDefinition`.  
**Client presentation owner:** `client_bevy::combat`; mouse/window/camera data and Bevy entities are never authoritative.

**Dependencies:**

- Code modules: `content_schema`, `sim_content`, `sim_core`, `client_bevy`.
- Content IDs: `class.grave_arbalist`, `ability.arbalist.primary_crossbow`, `item.prototype.weapon.pine_crossbow`, `arena.prototype.bell_laboratory_01`, `fp.1.0.0`.
- Previous task: `GB-M01-01B`.

**Exact source rules and prototype defaults:**

- `SIM-003`: `aim` is mouse position and independent from movement; `primary_fire` defaults to left mouse, allows hold, and remains rebindable.
- `SIM-004`: input is sampled every rendered frame, latest compact state is consumed at 30 Hz, primary presses carry sequence numbers, and captured menus block combat actions.
- `SIM-010`/`TECH-004`: projectiles and fire cadence are authoritative `sim_core` state; the client predicts/presents locally initiated attacks.
- `CLS-020`: one narrow bolt, 9.5-tile range, 2.2 attacks/second, 0.10-tile radius, and stop on first enemy unless pierce is granted.
- `CONT-010`: ordinary 455 ms interval rounds to 14 ticks at 30 Hz; `+x` is east, `+y` is screen-south.
- `CONT-013`: lifetime derives from range/speed and rounds up, producing 24 potential travel ticks for 9.5 tiles at 12 tiles/second.
- `CONT-FP-006`: Pine Crossbow has fixed damage 20, interval 455 ms, range 9.5, speed 12, radius 0.10, one bolt, and zero base pierce.
- The class primary record must agree on range, radius, 2.2 attacks/second, and stop-on-first-enemy. Any mismatch fails content compilation.

**Input and aim algorithm:**

1. Each rendered frame reads the configured primary mouse binding and cursor position.
2. The active `Camera2d` converts viewport cursor coordinates into centered render-world coordinates.
3. Convert to northwest simulation coordinates: `sim_x = render_x + arena_width/2`, `sim_y = arena_height/2 - render_y`.
4. Normalize `cursor_sim - player_sim` into `AimDirection`. Non-finite or zero-length values are rejected; the sampler retains the last valid direction. Initial aim is east `(1,0)`.
5. A false-to-true primary transition increments a checked `u32` press sequence exactly once. Held frames retain that sequence. Sequence overflow fails closed rather than wrapping.
6. If the combat-input gate is blocked, held fire is false and no shot may be created; simulation time continues.

**Fire and projectile tick order:**

1. Advance and range-clamp projectiles that existed at tick start, in ascending projectile ID.
2. Remove projectiles whose cumulative distance reached the exact authored range and emit expiry IDs.
3. Decrement the weapon interval timer when nonzero.
4. Validate the latest press sequence. A rising held edge requires a strictly newer sequence; a regressed/malformed sequence returns a typed error.
5. When held and the timer is zero, fire immediately using the tick's latest aim and current player position, allocate one stable simulation ID, emit one shot, then set the timer to 14.
6. A newly fired projectile remains at its origin until the next fixed tick. Releasing does not reset the timer. Re-pressing after expiry fires immediately; re-pressing before expiry waits for the first legal tick.
7. Each travel tick advances at most `12/30 = 0.4 tiles`; the final step clamps to the remaining distance, so position never exceeds exactly 9.5 tiles. Direction never changes after release.

**Legal state transitions:**

- Input: `Released(seq N) -> Pressed(seq N+1) -> Held(seq N+1) -> Released(seq N+1)`.
- Weapon: `Ready -> Fired/Cooling(14) -> ... -> Cooling(1) -> Ready`; held Ready fires on that same tick.
- Projectile: `Spawned -> Travelling -> ExpiredAtRange`; collision terminal states are not legal until `GB-M01-02B`.
- Runtime: `Boot -> ValidateContent -> CompileWeapon -> SpawnCombatPresentation -> Running -> Stopped`.

**Rejected inputs and typed errors:** missing/duplicate/wrong-operation item effects, negative/nonrepresentable values, class-primary mismatches, invalid weapon constants, zero/non-finite aim, stale or missing press sequence on a rising edge, sequence overflow, and entity-ID exhaustion fail closed. Unknown item effects remain legal on the record but cannot substitute for required exact fields.

**Network/version behavior:** no network transport in M01. The latest-state input boundary includes normalized aim, held state, and press sequence so the later 30 Hz datagram can use it without moving gameplay logic into Bevy.

**Persistence/idempotency:** none. Projectile IDs and press sequences are ephemeral per LocalLab run. One accepted press transition increments once even when sampled across many render frames.

**UI states:**

- Loading: no interactive frame before content and weapon compilation succeeds.
- Ready: reticle/crossbow visible; HUD shows Pine Crossbow values and `READY`.
- Held/cooling: HUD shows remaining ticks; muzzle flash appears only for an emitted shot.
- Blocked: primary action neutral; simulation and projectile travel continue.
- Error: startup failures return typed context before opening a window; impossible runtime invariants fail loudly in LocalLab.
- Empty/disconnected/retry: not applicable before networking.

**Telemetry events/fields:** no product telemetry before `GB-M01-10B`. Debug logs: feature ID, weapon/content IDs, compiled interval/lifetime ticks, press sequence, shot ID/tick/origin/direction, and expiry ID/tick.

**Accessibility/readability:** owner bolts use the GDD's physical-projectile bone-white pointed language and must not use hostile-red priority. Reticle and crossbow orientation use shape and luminance, not color alone. Persistent HUD stays outside the center 60% aiming corridor.

**Tests required:**

- Unit: duration rounding, content extraction, class-primary cross-checks, aim normalization/rejection, press transitions, immediate first shot, 14-tick held cadence, release/re-press behavior, stable IDs, 0.4-tile travel, exact 9.5-tile clamp, and 24-tick expiry.
- Deterministic: a fixed aim/fire trace matches exact IDs, shot ticks, projectile positions, and a checked-in bit snapshot in repeated processes/platform CI.
- Integration: validated `fp.1.0.0` compiles the exact Pine Crossbow; Bevy defaults to left mouse and converts render/simulation axes bidirectionally.
- Adversarial/failure: stale/missing sequences, zero/non-finite aim, malformed/duplicate item effects, wrong primary values, sequence/ID exhaustion, no cursor, blocked input, and moving/camera-following aim cannot create unintended shots.
- Visual/manual: launch optimized LocalLab; verify crossbow/reticle/bolt/HUD hierarchy and engine-native frame capture.
- Regression: firing cannot alter movement state; camera location cannot alter a given cursor world target's simulation result; collision remains absent rather than partially implemented.

**Feature flag/kill switch:** none for LocalLab. The combat-input gate is not a product kill switch.  
**Rollback/migration behavior:** revert feature commits. Changes to tick order, cadence, aim encoding, projectile motion, or compiled item semantics require ADR/task updates and regenerated golden fixtures.

## Acceptance criteria

1. Validated content compiles exactly one Pine Crossbow definition with damage 20, 14-tick interval, 9.5 range, speed 12, radius 0.10, count 1, pierce 0, and stop-on-first-enemy.
2. Rebind-ready mouse input produces independent, sequence-stable aim/held-fire actions; holding fires immediately and then exactly every 14 ticks without movement or render-frame dependence.
3. Stable-ID projectiles travel on their locked direction, advance 0.4 tiles per full tick, clamp to exactly 9.5 tiles, and expire after the 24th travel tick without collision/damage side effects.
4. The client visibly presents aim, crossbow, muzzle feedback, bolts, and diagnostics while camera/input presentation cannot mutate authoritative movement/combat state.
5. Unit, deterministic, content, adversarial, visual, full local CI, optimized Windows runtime, and clean GitHub CI checks pass.
