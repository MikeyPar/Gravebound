# GB-M03-02A — PostgreSQL architecture and wipeable identity schema

**Status:** In implementation under approved `SPEC-CONFLICT-005` and accepted `ADR-029`.

## Objective

Create the reusable M03 PostgreSQL foundation and only the relational schema already justified by `GB-M03-01`. Local and CI gates must exercise real PostgreSQL, detect schema drift, and fail explicitly when no database route exists.

## Three design authorities

1. `Gravebound_Production_GDD_v1_Canonical.md`: `TECH-001` through `TECH-006`, `TECH-020`, `TECH-021`, `TECH-030`, and `TECH-060`.
2. `Gravebound_Content_Production_Spec_v1.md`: stable/versioned content identifiers, idempotency, provenance boundaries, and no premature Core promotion.
3. `Gravebound_Development_Roadmap_v1.md`: M03 dependency model, persistence ADR requirement, `GB-M03-02`, and `GB-M03-11`.

## Scope

- Workspace `persistence` crate with asynchronous PostgreSQL pool, redacted configuration, embedded migrations, readiness/schema compatibility report, and typed errors.
- Pinned SQLx and PostgreSQL versions recorded in `ADR-029`.
- Checked-in wipeable namespace, account, character, selected-character, and bounded mutation-result schema.
- Relational length, capacity, ownership, version, and foreign-key backstops without moving domain reducers into SQL.
- Docker Compose test route with generated ephemeral password and disposable storage.
- Equivalent explicit `TEST_DATABASE_URL` route.
- Mandatory PostgreSQL CI job and fail-closed local command.

## Exclusions

- Durable repository adapter and server restart journey (`GB-M03-02B`).
- Item/vault, death/memorial/Echo, currency/ledger, Hall, or production namespace schema.
- SQLite, generic account JSON, SQL-owned product rules, stored credentials, or silently skipped database evidence.

## Acceptance criteria

1. Unit tests prove configuration validation and credential redaction; warnings-denied all-target Clippy passes.
2. Real PostgreSQL applies the migration twice safely, reports exact schema version and wipeable namespace, exposes exactly the approved tables, and enforces representative bounds.
3. `persistence-ci` uses `TEST_DATABASE_URL` or disposable Docker and otherwise fails with an actionable prerequisite; GitHub CI always executes the ignored-by-default database target against pinned PostgreSQL.
4. Workspace/content gates remain green, migration bytes use stable LF checkout, documentation and feature registry are current, and no later M03 domain table exists.
