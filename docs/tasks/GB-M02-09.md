# GB-M02-09 — shared authoritative combat arena

## Authority and objective

This task is subordinate to the Canonical Production GDD, Content Production Specification, and Development Roadmap. It resolves the literal M02 requirement that four humans complete one combat test together without pulling M03 persistence or M04 party scaling forward.

| Field | Contract |
|---|---|
| Objective | Four authenticated native clients play in one server-owned `fp.1.0.0` combat world, observe the same enemies and projectiles, and retain independent life, input, inventory, pickups, and terminal outcomes. |
| Simulation owner | One `SharedAuthoritativeArena` per hosted instance in `sim_core`; the instance scheduler advances it exactly once per 30 Hz frame. |
| Endpoint owner | A logical session owns transport binding, ingress sequence state, latest input, reliable edges, idempotency cache, snapshot sequence, and one stable player entity binding. It does not own an enemy world. |
| Content | Immutable `fp.1.0.0`; no health scaling, party bonus, persistence, hub, realm, dungeon, or M03/M04 content. |
| Population | One to four players before activation. A fifth admission allocates another instance. No late join enters an active encounter. |
| Promotion rule | Existing isolated-session evidence is regression evidence only and cannot close the four-human shared-combat row. |

## Deterministic data model

`SharedAuthoritativeArena` owns one arena geometry, tick, state version, normal-wave enemy set, enemy-health set, hostile projectile set, lane set, global entity allocators, encounter phase, and `BTreeMap<EntityId, ArenaPlayer>`.

Each `ArenaPlayer` owns:

- stable player entity ID, movement, combat, vitals, and per-player hostile damage-grace deadline;
- latest frozen input for the current tick;
- inventory, eligibility, personal pickups, reward ordinal, and projectile provenance;
- `Connected`, `LinkLost`, `Recalled`, or `Dead` lifecycle facts needed by simulation;
- automatic Recall deadline when link is lost. Manual Recall is unavailable in `fp.1.0.0`.

Player entity IDs begin at `10_000`, increase monotonically, and are never reused within an arena. Reconnect retains the same ID. Friendly projectile IDs are globally unique inside the arena. Snapshots expose projectile source player identity in addition to input sequence and projectile ordinal.

## Exact tick transaction

Every step clones the complete aggregate and commits only after all operations succeed:

1. Freeze one latest input per nonterminal player in ascending player entity ID. LinkLost input is neutral.
2. Build one collision world from the current living enemies.
3. Step each living player's movement and combat in ascending player ID.
4. Merge friendly collisions and raw damage intents, then sort by `(projectile_id, contact_ordinal)`.
5. Apply the combined combat step to shared enemy health exactly once.
6. Select each target-seeking enemy's living target by shortest squared distance; exact ties choose the lower player entity ID.
7. Advance living enemies in existing spawn-instance order and spawn threats from authored events only.
8. Resolve each active lane against every living player in ascending player ID, at most once per `(cast_id, player_id)`.
9. Advance every hostile projectile once in projectile-ID order against all living player hurtboxes and solids. Earliest swept contact wins; exact contact ties choose the lower entity ID. Nonpiercing shots stop on first contact. Piercing shots may hit each player once and retain the existing ignored-player set.
10. Apply hostile damage with a separate three-tick grace deadline per player.
11. Materialize one personal copy of every due reward for every eligible living participant. Pickup IDs are unique and owner-bound.
12. Resolve player death, then automatic LinkLost Recall. Death on the Recall deadline wins.
13. Mark shared combat complete only when all authored enemies are dead. A player terminal outcome never clears shared threats while another player remains active.
14. Increment state version and commit the clone.

## Player-local terminal rules

- Death clears only the dead player's friendly projectiles, pending/unsecured inventory, and personal pickups. It cannot clear shared enemies, hostiles, lanes, or another player's state.
- Link loss neutralizes only that player's input and leaves the character vulnerable for exactly 90 simulation ticks.
- Reconnect before final resolution rebinds the same entity and resumes ordinary input.
- Automatic Recall clears only that player's unsecured state and routes only that session to Lantern Halls.
- Survivors continue the shared encounter. The hosted arena closes only after every participant is terminal or completion delivery is acknowledged and all routes are released.

## Protocol 1.5 contract

- Accepted Join and Reattach results include a nonzero `controlled_entity_id`; unbound/rejected/Leave results do not.
- The client binds prediction to that entity before snapshot assembly and fails closed on late or zero rebinding.
- Friendly projectile snapshots must add `source_entity_id` before shared combat ships, preventing a remote player's coincident input sequence from confirming a local predicted shot.
- `RecallStart` and `RecallCancel` return `RecallUnavailableCombatLaboratory` without changing gameplay state.
- Automatic LinkLost Recall is server-internal and remains available.
- Each recipient snapshot contains all shared players, enemies, and projectiles, only that recipient's personal pickups, and only that recipient's acknowledged input sequence.

## Implementation sequence

## Acceptance tests

1. Two to four players damage one enemy and observe identical health and death tick.
2. Permuting network arrival order produces the same canonical state hash.
3. Target ties choose lower player ID; lane contact is once per player/cast.
4. Nonpiercing and piercing hostile shots resolve correctly across multiple players with per-player grace.
5. One player dies or automatically Recalls while survivors and shared threats continue unchanged.
6. Death on the LinkLost deadline beats Recall; reconnect beforehand restores the same entity.
7. Every eligible survivor receives an independently collectible personal pickup; forged, cross-owner, and duplicate requests fail.
8. Four clients receive all four players and the same shared enemies/projectiles, but recipient-specific acknowledgements and pickups.
9. A remote projectile cannot confirm a local predicted projectile.
10. Fifth admission creates another arena and late active-instance admission fails closed.
11. Existing one-player deterministic traces remain unchanged.
12. Impairment and abuse gates pass; sixteen bots run as four shared instances with zero stalls, divergence, residue, or monotonic post-warmup memory growth.
